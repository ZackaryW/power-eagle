# Adopt the design-v2 convention-based plugin runtime and drop v1 compatibility for the rewrite.

## Context
- The attached design-v2 document defines a full rewrite rather than an incremental extension of the legacy plugin.json and main.js or main.py model.
- The repository currently carries legacy memory describing the older plugin(context) plus powersdk runtime alongside new source work that is building a peagle-based runtime.
- That split created contradictory active memory about plugin packaging, entrypoints, runtime APIs, and development workflow.

## Content
- v2 is the authoritative target architecture for the rewrite and intentionally drops v1 compatibility.
- Plugin packages move from plugin.json plus one script entrypoint to a convention-based folder contract centered on plugin.js, ui.json, state.js, optional migration.js, actions, data, and render files.
- The runtime contract moves from plugin(context) with eagle and powersdk to a peagle runtime with local, ext, eagle, slot, and state.batch surfaces.
- Plugin rendering becomes declarative and slot-driven, while shared reusable logic moves into a global ext preset registry.
- Bucket-based distribution, URL install, flat module registry loading, template caching, and state migration are part of the v2 baseline rather than optional later additions.

## Consequences / notes
- Legacy memory describing plugin.json, main.js or main.py entrypoints, and plugin(context) authoring is historical and should no longer remain the active model for rewrite decisions.
- Rewrite work should be evaluated against design-v2 first, then reconciled with oldref only as behavioral archaeology rather than as the active contract.
- zmem updates for plugin architecture, setup, and runtime behavior should now prefer v2 terminology and flows.
