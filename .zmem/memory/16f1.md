# Never hardcode plugin code in logic files; keep plugin implementations in separate source files and load them dynamically.

## Context
- The code quality notes treat separation of plugin content from runtime logic as a non-negotiable maintainability rule.
- Mixing plugin examples or executable plugin bodies into manager logic would tightly couple runtime code to sample content.

## Content
- Plugin implementations belong in plugin source files or example modules, not embedded in discovery, loading, or management logic.
- The runtime should load plugin code from files or explicit example modules instead of keeping inline plugin bodies.
- This keeps the system extensible and prevents runtime components from becoming a dumping ground for sample behavior.

## Consequences / notes
- New examples and templates should be added in dedicated plugin-facing files.
- If runtime logic starts accumulating embedded plugin behavior, that is a signal the boundary is drifting.
