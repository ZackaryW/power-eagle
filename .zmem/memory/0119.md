# Plugin runtime modules load as native ESM with no bundler or transpilation step, relying on Eagle's Electron environment for Node built-ins at runtime.

## Context
- The design-v2 document explicitly rejects Vite or other build tools as part of plugin activation because plugins are downloaded as zips and must run immediately after extraction.
- The Eagle host environment provides Electron and Node capabilities, but those capabilities belong to the plugin runtime context, not to a generic browser bundle.

## Content
- plugin.js, actions, data, state.js, and migration.js are loaded as native ES modules during activation.
- The runtime uses a flat registry populated up front so peagle.local lookups do not load modules lazily at call time.
- Node built-ins such as fs, path, and process remain valid inside the Eagle-hosted runtime where Electron exposes them.
- Plugin activation must not assume a separate bundling or transpilation phase before installation.

## Consequences / notes
- Host application code should preserve the boundary between browser-safe UI code and Eagle-hosted runtime code.
- Bugs caused by bundling runtime-only paths into browser-only surfaces indicate a violation of the v2 import boundary.
- Plugin tooling and examples should assume native ESM and extracted file execution instead of a Vite-driven plugin workflow.
