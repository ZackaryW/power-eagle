# v2 hosts local-bucket, bucket-sourced, and URL-installed plugins through a native-ESM activation pipeline with a flat local registry, slot-based rendering, and shared ext presets.

## Context
- The active direction replaces the older manager-first runtime description with a host that indexes plugins from bucket folders, installs local plugins into the local bucket, and activates them through a convention-based package loader.
- The old architecture node centered on discovery, loading, execution, management, and Python evaluation around the legacy runtime contract.

## Content
- Power Eagle v2 remains a React-based Eagle extension host, but its plugin model is now package-oriented rather than plugin.json plus one script file.
- The local bucket at `~/.powereagle/buckets/local/plugins/<pluginId>` is the canonical on-disk home for locally installed plugins.
- Bucket-sourced plugins and direct URL installs converge on the same activation sequence: materialize a plugin folder under the host-managed bucket layout, read manifest, parse templates, load modules into a flat registry, render ui.json, and execute plugin.js.
- Plugin runtime behavior is organized around the peagle namespaces: local for this plugin, ext for shared presets, eagle for host API wrappers, slot for named UI slots, and state.batch for coalesced reactive renders.
- The plugin index UI is organized around installed, buckets, and url install tabs, with installed plugins launched from the host UI rather than acting as the host itself.
- Shared template parsing, state persistence and migration, ext presets, and activation-time registry loading are part of the core runtime rather than plugin-specific utilities.

## Consequences / notes
- Architecture work should now treat the flat registry loader, declarative UI renderer, and peagle runtime as core subsystems.
- Discovery and installation remain important, but the current host model uses bucket folders directly as the runtime source of truth instead of a separate `extensions/` install surface.
- Old manager-based notes remain useful for historical behavior and filesystem flow, but they are no longer the active architectural source of truth for the rewrite.
