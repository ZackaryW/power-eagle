# Run each plugin in an isolated execution context with plugin-scoped DOM, storage prefixes, and tracked cleanup.

## Context
- Isolation is both a runtime safety requirement and a repeatable implementation pattern in the current architecture.
- The legacy notes call out DOM ownership, storage namespacing, and event cleanup as the practical mechanisms for isolation.

## Content
- Each plugin owns a dedicated container rather than writing into shared page structure blindly.
- Storage keys are prefixed by plugin identifier to avoid collisions.
- Event listeners and related resources are tracked so they can be removed when plugins switch or unload.

## Consequences / notes
- New UI and runtime integrations should declare ownership boundaries clearly.
- Isolation should be preserved even when adding richer plugin interactions or long-lived listeners.
