# PythonScriptRunner executes Python plugins while PythonScriptEvaler intercepts callback signals and performs validated Eagle API calls.

## Context
- Python support is one of the major differentiators of the current platform, especially after the callback work.
- The legacy documentation distinguishes between running Python code and evaluating callback signals emitted during execution.

## Content
- PythonScriptRunner is responsible for launching Python scripts with Eagle context integration.
- PythonScriptRunner passes Eagle context through POWEREAGLE_CONTEXT and coordinates output streaming and execution lifecycle.
- PythonScriptEvaler processes stderr callback signals, filters them out of normal output, validates the token, and executes the requested Eagle API call safely.
- The evaluator integrates with the stderr handling path so original stderr handlers only receive filtered non-callback lines.
- The Python execution path now reduces overhead by caching callback prefixes during stderr handling, buffering stdout and stderr chunks before join, and batching output scroll updates with requestAnimationFrame.
- This relationship creates bidirectional communication while keeping execution and callback processing as distinct responsibilities.

## Consequences / notes
- Callback features should be implemented through the evaluator boundary rather than by expanding the runner into a second API execution layer.
- Security-sensitive behavior, especially token validation and safe execution, belongs close to the evaluator.
- Output semantics matter here: callback lines are meant to disappear from final stderr while ordinary Python errors remain visible.
- Long-running Python plugins should benefit from these optimizations without changing their author-facing contract.
