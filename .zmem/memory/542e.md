# Build plugin capabilities through createPowerSDKContext and organized namespaces such as visual, utils, storage, and webapi.

## Context
- Plugin capabilities are intentionally grouped so authors can discover features without memorizing scattered imports.
- The memory-bank describes createPowerSDKContext as the central builder for a consistent SDK surface.

## Content
- The SDK builder assembles namespaced capabilities including storage, container access, visual components, utility helpers, plugin identity, and web API access.
- The visual namespace covers higher-level UI primitives such as Button, Dialog, and CardManager.
- The utils namespace groups file, path, DOM, and common helpers instead of scattering standalone helpers across the runtime.
- Namespaces such as visual and utils provide a stable organization layer over implementation details.
- Centralized context construction keeps plugin runtime behavior consistent across all plugins.
- Core SDK modules are now imported statically during context construction to avoid repeated dynamic import overhead on plugin context creation.
- The SDK surface now names the host Eagle runtime separately from the webapi HTTP client so runtime-provided capabilities and web API calls are not conflated.

## Consequences / notes
- SDK additions should extend the organized namespace model rather than exposing ad hoc globals.
- This pattern improves discoverability and reduces drift between plugin types and examples.
- Example code and reference material should keep teaching the namespaced API surface because it reflects the intended authoring model.
- Performance work in SDK assembly should preserve the stable namespaced interface while reducing per-context setup overhead behind it.
- Type exports should preserve the host-versus-webapi split so SDK consumers can model the two identities correctly.
