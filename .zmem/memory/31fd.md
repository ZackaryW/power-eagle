# v2 plugin development uses installed-style folders with plugin.js, ui.json, state.js, actions, data, and render files, and manual validation inside Eagle rather than plugin bundling.

## Context
- The v2 design rejects bundler-driven plugin development and keeps plugin runtime expectations close to the extracted on-disk package shape.
- Older setup notes described plugin.json plus main.js or main.py inside the Power Eagle extensions directory, which no longer matches the v2 file contract.

## Content
- Plugin authors develop against the same installed-style folder structure that Power Eagle activates after extraction.
- Manual validation still happens inside Eagle, because host APIs, Node built-ins, and runtime UI behavior depend on Eagle's Electron environment.
- Local testing guidance should focus on exercising the plugin through the host panel, especially for slot wiring, host API wrappers, migration, and install flows.
- Repository build tooling remains for the host application itself, but plugin packages are not expected to be bundled by Vite before use.

## Consequences / notes
- Setup docs should distinguish clearly between building the Power Eagle host application and validating a plugin package inside Eagle.
- Development guidance should describe the v2 folder contract directly, not the retired plugin.json or main.js layout.
- Realtime manual checking remains a host-runtime process and should not be conflated with browser-only Vite previews.
