# Manager modules follow single-responsibility boundaries for discovery, loading, execution, management, download, and Python evaluation.

## Context
- The manager split is one of the clearest structural rules in the legacy system notes.
- Separate modules exist to make plugin lifecycle behavior easier to reason about and change safely.

## Content
- Discovery handles scanning and locating plugins.
- Loading resolves plugin code and examples.
- Execution runs JavaScript plugins with context injection.
- Management handles removal, hiding, and cleanup.
- Download owns archive validation, extraction, and installation.
- Python execution is isolated in dedicated runner and evaluator components.
- Performance-sensitive manager paths now prefer async filesystem work and parallelizable batches over synchronous serial processing.

## Consequences / notes
- Features should be added in the narrowest responsible module first.
- When responsibilities begin to blur, refactoring pressure should favor re-establishing the manager boundaries.
- Hot-path optimizations should usually stay inside the responsible manager instead of introducing cross-manager coupling.
