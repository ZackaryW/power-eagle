# Plugins use a universal plugin(context) interface exposing eagle and powersdk namespaces.

## Context
- The project deliberately standardizes the plugin entry contract to reduce author confusion and internal branching.
- The legacy notes present the universal context object as the main way plugins access Eagle and SDK capabilities.

## Content
- JavaScript plugins are expected to export a plugin(context) function.
- The context exposes the host-provided global eagle runtime plus the organized powersdk surface.
- Typical plugin implementations immediately destructure { eagle, powersdk } and work through powersdk.container, powersdk.storage, visual components, and Eagle APIs or notifications.
- The host-provided eagle runtime is a different identity from powersdk.webapi, which is the SDK's HTTP client surface.
- This single interface keeps examples, runtime injection, and documentation aligned.

## Consequences / notes
- New runtime features should be exposed through the existing context surface instead of inventing alternate plugin entry contracts.
- Consistency here lowers authoring complexity and keeps plugin examples interchangeable.
- Example and reference material should keep teaching the destructured context pattern because it matches the intended runtime model.
- Documentation should keep the two API identities distinct so plugin authors do not confuse host runtime methods with HTTP webapi calls.
