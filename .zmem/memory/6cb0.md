# Plugins expose a static manifest plus a default peagle entrypoint, with declarative ui.json and behavior split across actions, data, state, and render files.

## Context
- The v2 design explicitly separates declarative UI, imperative logic, and pure data so plugins are easier to generate, test, and evolve.
- The old plugin(context) plus powersdk pattern bundled UI construction, event wiring, and business logic more tightly inside one executable module.

## Content
- plugin.js exports a static manifest object and a default async entrypoint that receives peagle as its only argument.
- ui.json declares the UI tree and wires user interactions through namespace tuples rather than querySelector and manual addEventListener calls.
- state.js is a reactive object whose mapped keys trigger slot renders, while actions mutate state and call peagle surfaces.
- data files stay pure and directly imported by actions rather than being resolved through peagle.local.
- render templates are separate JSON assets that bind item data into reusable list or card structures at render time.

## Consequences / notes
- Runtime and example code should not drift back toward giant plugin modules that rebuild DOM imperatively.
- New plugin examples should demonstrate the peagle surfaces and file split rather than legacy powersdk container patterns.
- Circular behavior wiring should be handled through the flat registry and func or invokeFunc semantics rather than ad hoc imports across action files.
