# The shared file-creation preset is file.createWithContent; do not reintroduce document.createTextFile or document.createTextFileFromState aliases.

## Context

The oldref file creator flow is plugin-owned first and Eagle-backed second: a user action enters plugin logic, the plugin decides the effective file name, extension, and default content, and only then delegates the actual save/write/notify work to the shared Eagle-facing helper.

That requirement applies to file-creator because it owns extension-list behavior. It does not require every simpler example to add a local action file when the same behavior can be expressed declaratively.

## Content

Keep file.createWithContent as the only shared preset for file creation.

File Creator keeps the richer plugin-owned action flow: read state, normalize extension, persist a newly typed extension when appropriate, refresh extension cards, decide default content, then call file.createWithContent.

Simple Create Markdown keeps the stripped-down declarative form: one name input, fixed md extension, markdown content, no extension cache, and a direct plugin.json call to file.createWithContent. It stays independent and JSON-only.

The shared file.createWithContent implementation must follow the Eagle-native import path rather than a save dialog flow: take the resolved file name, determine content, write the file into a temporary path, call the Eagle global item.addFromPath API, then notify.

Do not hardcode a temp directory path in the host bridge or preset flow. Resolve it from Eagle os.tmpdir() when available, otherwise from the runtime host's real os.tmpdir() capability.

## Consequences / notes

Do not reintroduce document.createTextFile aliases.

The shared preset remains the Eagle-facing boundary for temp-file write, item.addFromPath import, and notification behavior. Use plugin-owned actions only where the plugin has real behavior to own above that boundary, such as file-creator's extension management.
